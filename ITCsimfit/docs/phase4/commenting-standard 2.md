# ITCsimfit Phase 4: 开发注释规范

## 1. 目的与适用范围

本规范用于 `ITCsimfit` 的新增/改动代码，目标是让注释可执行、可检查、可审阅。
本阶段执行策略：仅对新增/改动文件强制，不要求一次性重写全量历史文件。

## 2. 注释类型与必填字段

### 2.1 模块头注释（必填）

适用：每个模块文件顶部（例如 `R/*.R`、`R/server/body/**/*.R`）。

必须包含以下字段（原样字段名）：

1. `模块职责`
2. `依赖`
3. `对外接口`
4. `副作用`
5. `变更历史`

推荐模板：

```r
# [COMMENT_STD][MODULE_HEADER]
# 模块职责：...
# 依赖：...
# 对外接口：...
# 副作用：...
# 变更历史：YYYY-MM-DD - ...
```

### 2.2 关键算法注释（按需但建议在核心算法必填）

适用：复杂算法、数值稳定策略、回退路径。

必须包含以下字段（原样字段名）：

1. `算法目标`
2. `输入约束`
3. `数值稳定策略`
4. `失败回退`
5. `复杂度`

推荐模板：

```r
# [COMMENT_STD][KEY_ALGO]
# 算法目标：...
# 输入约束：...
# 数值稳定策略：...
# 失败回退：...
# 复杂度：...
```

### 2.3 I/O 契约注释（必填于边界函数）

适用：UI 输入整理、跨步骤 payload、导出打包、文件读写边界。

必须包含以下字段（原样字段名）：

1. `输入来源`
2. `字段/类型`
3. `单位`
4. `空值策略`
5. `输出保证`

推荐模板：

```r
# [COMMENT_STD][IO_CONTRACT]
# 输入来源：...
# 字段/类型：...
# 单位：...
# 空值策略：...
# 输出保证：...
```

### 2.4 错误语义注释（必填于统一错误入口）

适用：错误封装、异常传播、日志记录与用户提示边界。

必须包含以下字段（原样字段名）：

1. `错误码/类别`
2. `触发条件`
3. `用户可见性`
4. `日志级别`
5. `恢复动作`

推荐模板：

```r
# [COMMENT_STD][ERROR_SEMANTICS]
# 错误码/类别：...
# 触发条件：...
# 用户可见性：...
# 日志级别：...
# 恢复动作：...
```

## 3. 最小覆盖要求（Phase 4）

以下文件必须包含模块头注释样板，并至少覆盖一次对应专题注释：

1. `R/core_logic.R`：`[MODULE_HEADER]` + `[KEY_ALGO]` + `[IO_CONTRACT]`
2. `R/fitting.R`：`[MODULE_HEADER]` + `[KEY_ALGO]` + `[IO_CONTRACT]`
3. `R/server/body/runtime_core/01_bridge_state_inputs.R`：`[MODULE_HEADER]` + `[IO_CONTRACT]`
4. `R/server/body/snapshot_export/02_export_bridge.R`：`[MODULE_HEADER]` + `[IO_CONTRACT]`
5. `R/infrastructure/errors.R`：`[MODULE_HEADER]` + `[ERROR_SEMANTICS]`

## 4. 反例（不合格）

1. 只有“做了什么”没有“为什么”与失败策略。
2. 只写自然语言段落，未包含必填字段名。
3. I/O 边界函数不说明字段类型、单位和空值策略。
4. 错误处理不标注用户可见性和日志级别。

## 5. 审查与门禁

1. 代码评审中将检查注释标签与必填字段是否完整。
2. `tests/test_commenting_standard_minimal.R` 会执行最小结构检查。
3. 本阶段不启用自动重写，不做历史文件批量格式化。
